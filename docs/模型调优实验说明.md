# 煤炭配选大模型调优实验说明

## 1. 实验定位

本实验面向煤矿智能配煤系统中的“候选方案生成 + 方案解释生成”任务进行轻量领域适配。当前系统中，大模型已经参与候选配比方案生成，后端继续负责物料合法性校验、库存校验、质量预测和多目标评分排序。

这种定位与现有系统实现一致：大模型不是直接绕过后端给出最终方案，而是提出候选配比，系统再进行可行性与评分校验。该方式能够支撑任务书中“基于开源大模型煤炭配选垂直模型，输出可行配煤方案”的要求。

## 2. 基座模型

建议选择开源中文指令模型作为基座模型，例如：

- Qwen/Qwen2.5-1.5B-Instruct
- Qwen/Qwen2.5-3B-Instruct
- Qwen/Qwen2.5-7B-Instruct

本科毕设阶段优先选择 1.5B 或 3B，便于快速完成训练、评估和复现。

## 3. 数据来源

训练样本由主系统数据库快照和公开煤质资料共同构成，主要包含：

- 销售订单数据：需求量、灰分、硫分、水分、发热量、优先级、交付日期；
- 配煤方案数据：配比、预测煤质、成本、质量评分、成本评分、稳定性评分、综合评分；
- 规则知识数据：质量约束、库存约束、配比约束、经验规则；
- 历史案例数据：典型订单、历史配煤描述、执行效果；
- RAG 知识数据：规则、案例、术语和业务文档片段；
- 库存数据：煤种、仓库、可用量、批次信息。
- 公开煤质样本：水分、灰分、硫分、挥发分、发热量和来源说明。

数据构建命令：

```bash
python3 scripts/build_dataset.py
```

当前脚本会生成两类任务样本：

| 任务 | 输出格式 | 用途 |
|---|---|---|
| candidate_generation | `{"plans":[...]}` | 训练模型根据订单和候选物料生成候选配比 |
| plan_explanation | `{"ruleBasis":...}` | 训练模型对最终推荐方案生成结构化解释 |

样本规模以实际运行 `scripts/build_dataset.py` 的输出为准。

## 4. 调优目标

候选方案生成任务输入为订单、候选物料、规则、案例和 RAG 知识，输出必须为合法 JSON：

```json
{
  "plans": [
    {
      "planName": "低硫高热值候选方案A",
      "strategy": "优先满足低硫和热值约束，同时控制高价煤占比",
      "items": [
        {
          "coalId": 1,
          "productBatchNo": "PBxxx",
          "ratio": 0.6,
          "reason": "低硫、库存充足，作为主配物料"
        }
      ],
      "risk": "执行前需复核库存余量和煤质波动"
    }
  ]
}
```

方案解释任务输入为订单和推荐方案的完整上下文，输出必须为合法 JSON：

```json
{
  "ruleBasis": "规则依据",
  "caseReference": "案例参考",
  "recommendReason": "推荐理由",
  "riskTip": "风险提示",
  "finalExplanation": "最终解释"
}
```

调优目标以生成方案的业务效果为主，格式稳定性仅作为基础诊断指标。核心目标包括：

1. 提高预测煤质达标率，使生成方案在灰分、硫分、水分和发热量上更接近订单要求；
2. 提高质量安全余量，避免硫分、水分或热值长期贴近临界值；
3. 在满足质量约束的前提下降低吨煤成本；
4. 提高库存可执行性，减少生成方案中对库存不足物料的依赖；
5. 提高配比结构合理性，避免单一煤种占比过高或组合过度分散；
6. 提高风险控制能力，使风险说明能够反映质量余量、库存和煤质波动问题。

## 5. LoRA 训练方法

本实验采用 LoRA 参数高效微调。基座模型参数保持冻结，仅训练注意力层和前馈层中的低秩适配参数。建议初始参数如下：

| 参数 | 建议值 |
|---|---:|
| LoRA rank | 8 |
| LoRA alpha | 16 |
| LoRA dropout | 0.05 |
| epoch | 3 |
| learning rate | 2e-4 |
| max seq length | 2048 |

训练命令：

```bash
python3 scripts/train_lora.py --config configs/qwen_lora.yaml
```

## 6. 评估方法

模型优化前后对比以主系统后端评分为主。实验时分别启用优化前模型和优化后模型，让它们在同一订单、同一候选物料和同一业务规则下生成配煤方案；后端对模型生成的方案统一执行质量预测、成本计算、库存合理性评分和综合评分，并将结果写入 `experiment_record` 表。

这样对比的重点不是模型输出是否“格式友好”，而是模型生成方案在真实业务评价函数下的质量差异。

后端记录字段包括：

| 字段 | 含义 |
|---|---|
| experiment_code | 同一次生成实验的编号 |
| order_id | 订单ID |
| model_name | 本次方案生成使用的模型名称 |
| plan_id | 落库后的配煤方案ID |
| total_cost | 方案总成本 |
| avg_ash / avg_sulfur / avg_moisture / avg_calorific | 加权预测煤质 |
| quality_score | 质量匹配评分 |
| cost_score | 成本评分 |
| inventory_score | 库存合理性评分 |
| final_score | 综合评分 |
| constraint_hit | 命中约束、违反项、风险和评分说明JSON |
| risk_warning | 风险提示 |
| explain_text | 模型生成理由和评分解释 |

雷达图建议使用以下维度：

| 维度 | 对应字段 |
|---|---|
| 质量匹配 | quality_score |
| 成本优势 | cost_score |
| 库存合理 | inventory_score |
| 综合效果 | final_score |

生成接口可传入相同的 `experimentCode`，用于把优化前模型和优化后模型的结果归入同一组实验：

```json
{
  "orderId": 1,
  "candidateScope": "product_batch",
  "experimentCode": "EXP-BASE-VS-LORA-001"
}
```

后端接口：

```text
GET /experimentRecord/page
GET /experimentRecord/byOrder/{orderId}
GET /experimentRecord/radar?orderId=1
GET /experimentRecord/radar?experimentCode=EXP-O1-qwen-base-xxx
GET /experimentRecord/radar?orderId=1&modelName=qwen-coal-lora
```

离线验证集脚本仍可作为辅助诊断，用于检查模型输出结构和 JSON 稳定性：

| 指标 | 含义 |
|---|---|
| average_business_effect_score | 所有候选方案的平均业务效果综合分 |
| best_plan_business_effect_score | 每个订单中最优候选方案的业务效果均值 |
| 质量达标 | 预测灰分、硫分、水分、发热量满足订单约束的程度 |
| 质量余量 | 在达标基础上距离约束边界的安全余量 |
| 成本优势 | 生成方案相对于候选物料价格区间的吨煤成本优势 |
| 库存可执行 | 按订单需求量和配比计算后，物料库存是否足够 |
| 配比均衡 | 配比合计、煤种数量和主配比例是否合理 |
| 风险控制 | 风险说明是否覆盖质量、库存或煤质波动风险 |
| valid_json_rate | 基础诊断：模型输出能够被解析为合法 JSON 的比例 |
| valid_ratio_plan_rate | 基础诊断：候选方案 ratio 之和接近 1 的比例 |

评估命令示例：

```bash
python3 scripts/evaluate_json_outputs.py \
  --base-model Qwen/Qwen2.5-1.5B-Instruct \
  --limit 20 \
  --report-file outputs/reports/base_eval_report.json

python3 scripts/evaluate_json_outputs.py \
  --base-model Qwen/Qwen2.5-1.5B-Instruct \
  --adapter outputs/adapters/qwen-coal-lora \
  --limit 20 \
  --report-file outputs/reports/lora_eval_report.json
```

生成业务质量雷达图：

```bash
python3 scripts/plot_quality_radar.py \
  --base-report outputs/reports/base_eval_report.json \
  --tuned-report outputs/reports/lora_eval_report.json \
  --output outputs/reports/business_quality_radar.svg
```

论文中建议将 `valid_json_rate` 等结构指标作为“输出稳定性”辅助结果，将业务质量指标和雷达图作为模型优化前后对比的主体。

## 7. 论文表述建议

建议在论文中这样表述：

> 本文基于开源指令大模型构建煤炭配选领域适配模型。结合系统数据库中的订单、配煤方案、规则知识、历史案例和 RAG 知识片段，构造指令微调样本，并采用 LoRA 参数高效微调方法对模型进行领域化适配。微调后的模型不直接替代后端配煤优化算法，而是用于提升方案解释、规则依据生成、案例引用和风险提示的稳定性。系统最终形成“多目标评分寻优 + RAG 知识增强 + 领域适配大模型解释生成”的协同决策流程。
